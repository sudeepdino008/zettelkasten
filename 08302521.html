Appium

Introduction

  • Open source mobile app testing tool that is used for mobile automation testing.
  • It’s not a testing framework per se, but a framework for UI interaction. The Appium APIs for both Android and iOS are almost identical, which means if the applications are similar, your tests for the two will share most of the code.

Philosophy

open source

Shouldn’t need to recompile the app

Use vendor provided automation frameworks under the hood - UIAutomator/XCUITest

Shouldn’t be locked into one particular language to write the automation scripts.

Wrap the vendor-provided frameworks in one API - webdriver API. The WebDriver (Appium WebDriver) specifies a client-server protocol (JSON Wire protocol) to send appropriate requests to the server (more on this later).

Do not reinvent the wheel

Webdriver is a de facto standard for automating web browsers. Webdriver can simply be extended to include commands relevant for mobile apps, which is what happens for Appium.

interesting features

  • allows to test things like installing/uninstalling the app (could be useful to test login flows)
  • write once, run on both the mobile platforms.
  • works with cross platform languages (like flutter etc.)
  • Appium translates selenium webdriver commands into UIAutomation/XCUITest (iOS) or UIAutomator (Android) commands depending on device type.
  • Use browserstack to get the devices you want to test on (rather than relying on the simulators).

Key Concepts

  • Desired capabilities
    These are keys/values sent by clients to server when an automation session is requested. It provides info about what kind of session is being requested. It can be scripted in the WebDriver test or set within the appium server GUI (via an inspector session).
    The documentation is here; XCUITest specific capabilities are here (for example, autolaunch? can be configured)

Writing stable appium tests

link here

  1. Test flakiness might be due to any part of the automation stack (appium script, appium, WebDriver, XCUITest/UIAutomator, mobile operating system). Check where the issue exists.
  2. Use the correct Locator strategies
    There are several locator strategies. The preferred way is via accessibility identifiers–It’s cross platform (both iOS and android have concepts of accessibility labels). The only case it might be difficult is when appium is used by testers, and need to ask their developers to provide accessibility identifiers.
    There’s also predicate string strategy and class chain strategy (available as android or iOS specific methods).
    Xpath is to be completely avoided.
    Appium desktop recommends the locator strategy in its inspection session.
  3. Waiting for app states
    You might often not find the elements because of race conditions between the executing appium script accessing the element AND the UI being rendered on device.
    Strategies:
  1. Use Thread.sleep()–Meh!
  2. Use implicit waits, which keeps trying to find the element for the specified interval:
    driver.implicitly_wait(10)
  1. Explicit Wait: More fine grained control, allowing you to customize the timeout duration when locating each element.
  2. Custom explicit waits: For more complicated waits!

  1. Dealing with unfindable elements
    Custom UI elements might be visible on screen but might still not be accessible. This is a limitation because XCUITest or UIAutomator themselves cannot locate these custom elements. One solution is to enable accessibility (see for iOS and Android). The other solution is to do so by performing operations (tap, press etc.) on some coordinate.

  2. Setting up the app state Too much time might be spent on logging in. There are several ways to mitigate this:

  1. Custom android activities (for Android-only)
  2. Custom launch arguments (iOS only): iOS apps can be launched with command-line flags called launch arguments. App will then need to handle these launch arguments to do UI-less login.
  1. A testing screen: which can access the various parts of the app.
  2. Deep Linking

  1. Mocking external services
    User mock server to stub api responses. Might be problematic when you’re running the tests in cloud providers (like browserstack) – So see if it can be done there.

  2. Setting up iOS app permissions automatically

Webdriver

resource: python API for WebDriver
It’s the protocol that selenium and appium both use to interact with web browsers or mobile devices/simulators.
resource: w3c specification of webdriver

WebDriver is a remote control interface that enables introspection and control of user agents. It provides a platform- and language-neutral wire protocol as a way for out-of-process programs to remotely instruct the behavior of web browsers. Provided is a set of interfaces to discover and manipulate DOM elements in web documents and to control the behavior of a user agent. It is primarily intended to allow web authors to write tests that automate a user agent from a separate controlling process, but may also be used in such a way as to allow in-browser scripts to control a–possibly separate–browser.
Webdriver allows you to write test scripts in a number of programming languages. It controls the browser from OS level.
Before selenium webdriver (called selenium 2.0), there was selenium remote control (selenium 1.0), which is inferior to webdriver in several ways (link for details). Webdriver also allows headless (GUI-less) HTMLUnit browser which allows unit testing to be faster since no time is spend on waiting for the webpages to render.
